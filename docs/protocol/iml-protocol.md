# IML Protocol v2 Reference

The Instruction Markup Language (IML) is the structured protocol through which LLM agents communicate with LLMOS Bridge. Every interaction follows the same pattern: the agent produces a **plan** (a JSON document conforming to IML v2), submits it to the daemon, and receives structured results.

IML is not a programming language. It is a declarative specification that describes *what* the agent wants to accomplish, leaving the *how* to the daemon's orchestration engine.

---

## Philosophy

IML was designed around five principles:

1. **Declarative over imperative** — Plans describe desired outcomes, not step-by-step procedures. The daemon handles scheduling, error recovery, and resource management.

2. **Type safety at the boundary** — Every action parameter is defined as a Pydantic v2 model. The LLM gets JSON Schema documentation; the daemon validates before execution.

3. **Dependency-driven execution** — Actions declare dependencies on other actions via `depends_on`. The DAG scheduler extracts maximum parallelism while respecting ordering constraints.

4. **Graceful degradation** — Every optional field has a sensible default. A minimal plan with just `description` and `actions` is fully valid.

5. **Composability** — Plans can reference results of previous actions via templates (`{{result.X.Y}}`), memory via `{{memory.key}}`, and environment via `{{env.VAR}}`.

## Plan Structure

### IMLPlan

The top-level document submitted to `POST /plans`.

```json
{
  "plan_id": "download-and-process-7f3a",
  "protocol_version": "2.0",
  "description": "Download CSV from API, parse it, and insert rows into database",
  "session_id": "session-2024-01-15",
  "execution_mode": "parallel",
  "plan_mode": "standard",
  "metadata": {
    "created_by": "claude-agent",
    "llm_model": "claude-opus-4-20250514",
    "tags": ["data-pipeline", "csv"]
  },
  "module_requirements": {
    "api_http": ">=1.0.0",
    "database": ">=1.0.0"
  },
  "actions": [...]
}
```

| Field | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| `plan_id` | string | No | Auto-generated UUID | Unique identifier. Pattern: `[a-zA-Z0-9_-]+` |
| `protocol_version` | literal `"2.0"` | Yes | — | Protocol version. Must be `"2.0"` |
| `description` | string | Yes | — | Human-readable plan description |
| `session_id` | string | No | `null` | Groups related plans in a session |
| `execution_mode` | enum | No | `"sequential"` | `sequential`, `parallel`, or `reactive` |
| `plan_mode` | enum | No | `"standard"` | `standard` or `compiler` |
| `compiler_trace` | object | Conditional | `null` | Required when `plan_mode = "compiler"` |
| `metadata` | object | No | `null` | Non-functional metadata |
| `module_requirements` | object | No | `{}` | PEP-440 version specifiers per module |
| `actions` | array | Yes | — | 1 to 500 actions (configurable) |

### Execution Modes

| Mode | Behavior |
|------|----------|
| `sequential` | Actions execute one at a time in declaration order, respecting `depends_on` |
| `parallel` | Independent actions (no mutual dependency) execute concurrently in waves |
| `reactive` | Reserved for future use. Agent receives intermediate results and can submit follow-up plans |

### Plan Modes

| Mode | Behavior |
|------|----------|
| `standard` | Default. Plan generated by LLM probabilistically. No special requirements. |
| `compiler` | Plan generated through structured 4-phase reasoning. Requires `compiler_trace` with `generation_approved = true`. |

## Action Structure

### IMLAction

Each action in the `actions` array describes a single operation to be dispatched to a module.

```json
{
  "id": "download-csv",
  "action": "download_file",
  "module": "api_http",
  "params": {
    "url": "https://api.example.com/data.csv",
    "output_path": "/tmp/data.csv"
  },
  "depends_on": [],
  "on_error": "abort",
  "requires_approval": false,
  "timeout": 60,
  "label": "Download source data",
  "tags": ["download"]
}
```

| Field | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| `id` | string | Yes | — | Unique within plan. Pattern: `[a-zA-Z0-9_-]+` |
| `action` | string | Yes | — | Action name. Pattern: `[a-z][a-z0-9_]*` |
| `module` | string | Yes | — | Module ID. Pattern: `[a-z][a-z0-9_]*` |
| `params` | object | No | `{}` | Action parameters (supports templates) |
| `depends_on` | array | No | `[]` | IDs of actions that must complete first |
| `on_error` | enum | No | `"abort"` | Error handling policy |
| `requires_approval` | bool | No | `false` | Pause for user approval before execution |
| `timeout` | int | No | `300` | Execution timeout in seconds |
| `retry` | object | No | `null` | Retry configuration |
| `rollback` | object | No | `null` | Compensating action configuration |
| `perception` | object | No | `null` | Screenshot/OCR capture configuration |
| `memory` | object | No | `null` | Memory read/write configuration |
| `approval` | object | No | `null` | Custom approval configuration |
| `label` | string | No | `null` | Human-readable label |
| `tags` | array | No | `[]` | Metadata tags (max length per tag enforced) |

### Error Handling Policies

| Policy | Behavior |
|--------|----------|
| `abort` | Stop plan execution. Mark all downstream actions as `SKIPPED`. |
| `continue` | Log the error and proceed to the next independent action. |
| `retry` | Re-execute the action according to `retry` configuration. |
| `rollback` | Execute the compensating action defined in `rollback`. |
| `skip` | Skip this action and continue. Mark as `SKIPPED`. |

**Cascade failure**: When `on_error = "abort"`, the failure cascades to all transitive descendants in the dependency graph. Every action that directly or transitively depends on the failed action is marked `SKIPPED`.

### Naming Constraints

- **plan_id**: `[a-zA-Z0-9_-]+` (letters, digits, underscores, hyphens)
- **action.id**: `[a-zA-Z0-9_-]+`
- **action.action**: `[a-z][a-z0-9_]*` (lowercase start, then lowercase letters, digits, underscores)
- **action.module**: `[a-z][a-z0-9_]*`

These constraints are enforced by Pydantic validators at deserialization time.

## Template Engine

IML supports three template syntaxes for dynamic parameter resolution. Templates are resolved by the orchestration engine just before action dispatch.

### Result Templates

Reference the output of a previously completed action:

```
{{result.<action_id>.<field_path>}}
```

**Examples**:
```json
{
  "file_path": "{{result.download-csv.path}}",
  "content": "{{result.read-file.content}}",
  "ocr_text": "{{result.capture-screen._perception.after_text}}"
}
```

The field path supports dot-notation for nested access. If the referenced action has not completed or the field does not exist, template resolution raises an error.

### Memory Templates

Read values from the persistent key-value store:

```
{{memory.<key>}}
```

**Example**:
```json
{
  "api_key": "{{memory.github_token}}",
  "base_url": "{{memory.api_endpoint}}"
}
```

Memory values must be stored before the plan runs (via a previous plan's `memory.write_key` or the KV store API).

### Environment Templates

Read environment variables from the daemon process:

```
{{env.<VARIABLE_NAME>}}
```

**Example**:
```json
{
  "token": "{{env.GITHUB_TOKEN}}",
  "home": "{{env.HOME}}"
}
```

**Security note**: Environment templates are disabled in the `readonly` profile (`allow_env_templates = false`). The `local_worker`, `power_user`, and `unrestricted` profiles allow them.

### Template Resolution Order

Templates are resolved just before action dispatch, after all dependency actions have completed:

1. `depends_on` actions are verified as COMPLETED
2. `{{result.X.Y}}` templates are resolved from execution results
3. `{{memory.key}}` templates are resolved from KV store
4. `{{env.VAR}}` templates are resolved from environment
5. Resolved params are re-validated against sandbox paths
6. Action is dispatched to module

## Retry Configuration

When `on_error = "retry"`, the `retry` field controls retry behavior:

```json
{
  "retry": {
    "max_attempts": 3,
    "delay_seconds": 1.0,
    "backoff_factor": 2.0,
    "retry_on": ["ConnectionError", "TimeoutError"]
  }
}
```

| Field | Type | Default | Constraints | Description |
|-------|------|---------|-------------|-------------|
| `max_attempts` | int | 3 | >= 1 | Maximum retry attempts |
| `delay_seconds` | float | 1.0 | >= 0.1 | Base delay between retries |
| `backoff_factor` | float | 2.0 | 1.0 to 10.0 | Exponential backoff multiplier |
| `retry_on` | array | `[]` | — | Exception class names to retry on. Empty = retry on any error. |

**Delay calculation**: `delay = delay_seconds * (backoff_factor ^ attempt)`

If `on_error = "retry"` is set but no `retry` configuration is provided, a default `RetryConfig` is automatically created with `max_attempts=3, delay_seconds=1.0, backoff_factor=2.0`.

## Rollback Configuration

When `on_error = "rollback"`, a compensating action is executed:

```json
{
  "rollback": {
    "action": "cleanup-temp-files",
    "params": {
      "path": "{{result.download-csv.path}}"
    }
  }
}
```

| Field | Type | Description |
|-------|------|-------------|
| `action` | string | ID of the action to execute as compensation. Must exist in the same plan. |
| `params` | object | Override parameters for the rollback action. Supports `{{result.X.Y}}` templates. |

**Validation**: The `action` field must reference a valid action ID within the same plan. This is validated at plan construction time.

## Perception Configuration

Capture screenshots and OCR text around action execution:

```json
{
  "perception": {
    "capture_before": true,
    "capture_after": true,
    "ocr_enabled": true,
    "validate_output": "$.status",
    "timeout_seconds": 10
  }
}
```

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `capture_before` | bool | `false` | Screenshot before action executes |
| `capture_after` | bool | `false` | Screenshot after action completes |
| `ocr_enabled` | bool | `false` | Run OCR on captured screenshots |
| `validate_output` | string | `null` | JSONPath expression for output validation |
| `timeout_seconds` | int | 10 | Timeout for perception capture |

Perception data is stored under the reserved `_perception` key in the action's execution results:

```
{{result.action_id._perception.before_screenshot}}
{{result.action_id._perception.after_screenshot}}
{{result.action_id._perception.before_text}}
{{result.action_id._perception.after_text}}
```

## Memory Configuration

Read from and write to the persistent key-value store:

```json
{
  "memory": {
    "read_keys": ["api_endpoint", "auth_token"],
    "write_key": "downloaded_file_path",
    "vector_search": "similar documents about machine learning",
    "top_k": 5
  }
}
```

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `read_keys` | array | `[]` | KV keys to inject into params before execution |
| `write_key` | string | `null` | Store action result under this KV key |
| `vector_search` | string | `null` | Semantic query for ChromaDB vector search |
| `top_k` | int | 5 | Number of vector search results (1 to 100) |

**Read keys**: Values are injected into `params` under the same key name. If a key does not exist in the store, it is silently skipped.

**Write key**: After successful action execution, the entire result is serialized and stored under the specified key.

## Approval Configuration

Customize the approval experience for sensitive actions:

```json
{
  "approval": {
    "message": "This will delete all files in /tmp. Proceed?",
    "risk_level": "high",
    "timeout_seconds": 120,
    "timeout_behavior": "reject",
    "clarification_options": ["Yes, delete all", "Skip this step", "Modify path"]

}
```

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `message` | string | `null` | Custom approval prompt |
| `risk_level` | enum | `"low"` | `low`, `medium`, `high`, `critical` |
| `timeout_seconds` | int | `null` | Per-action timeout override |
| `timeout_behavior` | enum | `"reject"` | `reject` or `skip` on timeout |
| `clarification_options` | array | `[]` | Structured options for user decision |

## Compiler Mode

When `plan_mode = "compiler"`, the LLM must provide a structured reasoning trace demonstrating that the plan was generated through a disciplined 4-phase process.

```json
{
  "plan_mode": "compiler",
  "compiler_trace": {
    "analysis": "User wants to download a CSV file and insert it into a database. Two atomic intentions: (1) HTTP download, (2) database insertion.",
    "resolution": "Intention 1 maps to api_http.download_file. Intention 2 maps to database.insert_record with dependency on (1).",
    "validation": "Params validated: URL is well-formed, output_path is within sandbox, database table exists. Dependencies are acyclic.",
    "generation_approved": true,
    "llm_model": "claude-opus-4-20250514",
    "prompt_tokens": 1500
  }
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `analysis` | string | No | Phase 1: Decompose user intent into atomic intentions |
| `resolution` | string | No | Phase 2: Map intentions to module actions |
| `validation` | string | No | Phase 3: Verify params, dependencies, types |
| `generation_approved` | bool | Yes | Phase 4: Explicit approval. Must be `true`. |
| `llm_model` | string | No | Model that generated the trace |
| `prompt_tokens` | int | No | Token count for the generation prompt |

**Validation rule**: If `plan_mode = "compiler"`, the `compiler_trace` field must be present AND `generation_approved` must be `true`. Otherwise, the plan is rejected at parse time.

## Module Requirements

Plans can declare minimum module version requirements using PEP-440 version specifiers:

```json
{
  "module_requirements": {
    "filesystem": ">=1.0.0",
    "api_http": ">=1.0.0,<2.0.0",
    "database": "~=1.0"
  }
}
```

The executor validates these requirements against installed module versions before execution begins. If any requirement is not satisfied, the plan fails immediately with a clear error message.

Supported PEP-440 specifiers: `>=`, `<=`, `>`, `<`, `==`, `!=`, `~=`, `*`.

## Lifecycle States

### Plan Status

```
PENDING ──→ RUNNING ──→ COMPLETED
                   ├──→ FAILED
                   ├──→ CANCELLED
                   └──→ PAUSED
```

| Status | Description |
|--------|-------------|
| `pending` | Plan submitted but not yet started |
| `running` | At least one action is executing |
| `completed` | All actions completed successfully |
| `failed` | One or more actions failed (depending on error policy) |
| `cancelled` | Plan was cancelled via DELETE endpoint |
| `paused` | Plan is waiting for user approval |

### Action Status

```
PENDING ──→ WAITING ──→ RUNNING ──→ COMPLETED
                                ├──→ FAILED
                                ├──→ SKIPPED
                                ├──→ ROLLED_BACK
                                └──→ AWAITING_APPROVAL
```

| Status | Description |
|--------|-------------|
| `pending` | Not yet scheduled |
| `waiting` | Waiting for dependencies to complete |
| `running` | Currently executing |
| `completed` | Executed successfully |
| `failed` | Execution raised an error |
| `skipped` | Skipped due to cascade failure or skip policy |
| `rolled_back` | Compensating action was executed |
| `awaiting_approval` | Paused for user approval |

## Validation Rules

The protocol enforces these rules at plan construction (Pydantic validators):

### Plan-Level

1. `plan_id` must match pattern `[a-zA-Z0-9_-]+`
2. `protocol_version` must be `"2.0"`
3. `actions` must contain at least 1 action (max configurable, default 500)
4. All action IDs must be unique within the plan
5. All `depends_on` references must point to existing action IDs
6. All `rollback.action` references must point to existing action IDs
7. If `plan_mode = "compiler"`, `compiler_trace` must be present with `generation_approved = true`
8. Dependency graph must be acyclic (no circular dependencies)

### Action-Level

1. `id` must match pattern `[a-zA-Z0-9_-]+`
2. `action` must match pattern `[a-z][a-z0-9_]*`
3. `module` must match pattern `[a-z][a-z0-9_]*`
4. `timeout` must be within configured bounds
5. Tags must not exceed maximum length
6. If `on_error = "retry"` and no `retry` config provided, a default is auto-created

## Error Responses

When validation fails, the API returns structured error responses:

```json
{
  "error": "ValidationError",
  "detail": "Action 'download-csv' depends on non-existent action 'fetch-data'",
  "plan_id": "my-plan-123",
  "code": "INVALID_DEPENDENCY"
}
```

Common error codes:

| Code | Description |
|------|-------------|
| `VALIDATION_ERROR` | Plan structure does not conform to IML v2 |
| `INVALID_DEPENDENCY` | `depends_on` references non-existent action |
| `CIRCULAR_DEPENDENCY` | Dependency graph contains cycles |
| `PERMISSION_DENIED` | Security profile does not allow the action |
| `APPROVAL_REQUIRED` | Action requires user approval |
| `MODULE_NOT_FOUND` | Referenced module is not registered |
| `ACTION_NOT_FOUND` | Module does not expose the requested action |
| `VERSION_MISMATCH` | Module version does not satisfy requirement |
| `TEMPLATE_ERROR` | Template resolution failed |
| `SANDBOX_VIOLATION` | Path outside configured sandbox |

## Typed Parameters

Every module action has a corresponding Pydantic v2 model in `protocol/params/`. These models serve three purposes:

1. **Validation** — Input parameters are validated before action dispatch
2. **Documentation** — JSON Schema is auto-generated for LLM agent context
3. **Type Safety** — Python code gets typed access to parameters

**Example** — `filesystem.read_file`:

```python
class ReadFileParams(BaseActionParams):
    path: str = Field(..., description="Absolute path to the file")
    encoding: str = Field("utf-8", description="File encoding")
    offset: int | None = Field(None, description="Start line (0-indexed)")
    limit: int | None = Field(None, description="Maximum lines to read")
```

**Generated JSON Schema** (available at `GET /modules/filesystem/actions/read_file/schema`):

```json
{
  "type": "object",
  "properties": {
    "path": {
      "type": "string",
      "description": "Absolute path to the file"
    },
    "encoding": {
      "type": "string",
      "default": "utf-8",
      "description": "File encoding"
    },
    "offset": {
      "type": "integer",
      "description": "Start line (0-indexed)"
    },
    "limit": {
      "type": "integer",
      "description": "Maximum lines to read"
    }
  },
  "required": ["path"]
}
```

The full parameter reference for each module is available in the [Module Reference](../modules/reference/) section.

## IML Repair

When an LLM produces malformed JSON, the `IMLRepair` component attempts recovery:

1. **Fuzzy JSON parsing** — handles trailing commas, single quotes, unquoted keys, truncated output
2. **Schema-guided repair** — uses JSON Schema to infer missing required fields
3. **LLM correction loop** — if fuzzy parsing fails, formats a correction prompt for the LLM with the error details and expected schema

The repair pipeline is transparent to the caller — it runs automatically when standard JSON parsing fails.

---

## Protocol Migration

LLMOS Bridge includes a migration pipeline for converting plans from older protocol versions:

```
v1.0 Plan → MigrationPipeline → v2.0 Plan
```

The built-in v1.0 to v2.0 migration handles:
- Adding `protocol_version: "2.0"`
- Converting flat action lists to dependency-aware format
- Mapping legacy field names to v2 equivalents

Custom migrations can be registered for future version transitions.

## Complete Example

A full IML v2 plan demonstrating all major features:

```json
{
  "plan_id": "data-pipeline-001",
  "protocol_version": "2.0",
  "description": "Download sales data, process it, and update the database",
  "execution_mode": "parallel",
  "metadata": {
    "created_by": "sales-agent",
    "tags": ["pipeline", "daily"]
  },
  "module_requirements": {
    "api_http": ">=1.0.0",
    "filesystem": ">=1.0.0",
    "database": ">=1.0.0"
  },
  "actions": [
    {
      "id": "download-data",
      "action": "download_file",
      "module": "api_http",
      "params": {
        "url": "https://api.sales.com/export.csv",
        "output_path": "/tmp/sales.csv"
      },
      "on_error": "retry",
      "retry": {
        "max_attempts": 3,
        "delay_seconds": 2.0,
        "backoff_factor": 2.0
      },
      "timeout": 120,
      "label": "Download sales CSV"
    },
    {
      "id": "read-data",
      "action": "read_file",
      "module": "filesystem",
      "params": {
        "path": "{{result.download-data.path}}"
      },
      "depends_on": ["download-data"],
      "on_error": "abort",
      "label": "Read downloaded file"
    },
    {
      "id": "insert-records",
      "action": "execute_query",
      "module": "database",
      "params": {
        "connection_id": "sales-db",
        "query": "INSERT INTO daily_sales (data) VALUES (?)",
        "parameters": ["{{result.read-data.content}}"]
      },
      "depends_on": ["read-data"],
      "requires_approval": true,
      "approval": {
        "message": "About to insert sales data into production database. Proceed?",
        "risk_level": "medium"
      },
      "on_error": "rollback",
      "rollback": {
        "action": "cleanup",
        "params": {}
      },
      "memory": {
        "write_key": "last_import_result"
      },
      "label": "Insert into database"
    },
    {
      "id": "cleanup",
      "action": "delete_file",
      "module": "filesystem",
      "params": {
        "path": "{{result.download-data.path}}"
      },
      "depends_on": ["insert-records"],
      "on_error": "continue",
      "label": "Clean up temporary file"
    }
  ]
}
```

This plan demonstrates:
- **Dependency chaining**: `read-data` depends on `download-data`, `insert-records` depends on `read-data`
- **Template resolution**: `{{result.download-data.path}}` references output from a previous action
- **Retry with backoff**: `download-data` retries 3 times with exponential backoff
- **Approval gate**: `insert-records` requires explicit user approval
- **Rollback**: `insert-records` references `cleanup` as its compensating action
- **Memory persistence**: Result of `insert-records` is stored under `last_import_result`
- **Cascade semantics**: If `read-data` fails, `insert-records` and `cleanup` are both SKIPPED
- **Continue on error**: `cleanup` continues even if deletion fails
